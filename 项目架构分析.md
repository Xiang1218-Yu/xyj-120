# 生存系统项目架构分析

## 项目概述

本项目是一个「末日生存社区」Web 应用，采用 **React + TypeScript + Vite** 作为前端技术栈，**Express.js** 作为后端服务，是一个典型的前后端分离但同仓部署的全栈项目。

**技术栈：**
- 前端：React 18 + TypeScript + React Router v7 + Zustand + Tailwind CSS
- 后端：Express.js + TypeScript + CORS
- 构建工具：Vite 6
- 开发模式：前后端并行开发（concurrently），通过 Vite 代理转发 API 请求

## 一、接口封装方式

### 1.1 前端 API 封装层

前端统一封装在 [src/lib/api.ts](file:///Users/tog/Desktop/code/solo/xyj-120/src/lib/api.ts) 中，采用 **泛型函数 + 模块化组织** 的设计模式。

#### 核心请求函数

```typescript
const API_BASE = '/api'

async function request<T>(path: string, options: RequestInit = {}): Promise<T> {
  const response = await fetch(`${API_BASE}${path}`, {
    headers: {
      'Content-Type': 'application/json',
      ...options.headers,
    },
    ...options,
  })

  const data = await response.json()
  
  if (!response.ok || !data.success) {
    throw new Error(data.error || '请求失败')
  }

  return data.data as T
}
```

**设计特点：**
- 统一的基础路径 `/api`，配合 Vite 代理转发
- 自动设置 `Content-Type: application/json` 头
- 统一的响应格式校验：检查 `success` 字段
- 统一的错误抛出机制：HTTP 状态码非 2xx 或 `success: false` 时抛出 Error
- TypeScript 泛型支持，保证类型安全

#### 模块化 API 组织

所有 API 按业务领域组织，导出统一的 `api` 对象：

## 二、拦截器与错误处理机制

### 2.1 前端拦截层

前端请求封装函数 `request` 本身承担了拦截器的职责：

**请求拦截：**
- 自动添加 `Content-Type: application/json` 头
- 统一拼接 `/api` 基础路径

**响应拦截：**
- 统一解析 JSON 响应
- 检查 HTTP 状态码和业务状态码 `success`
- 统一抛出错误，由调用方 `try-catch` 处理

**错误处理示例（来自 [Community.tsx](file:///Users/tog/Desktop/code/solo/xyj-120/src/pages/Community.tsx#L146-L151)）：**

```typescript
try {
  const result = await api.community.create(postData)
  // 成功逻辑
} catch (error: any) {
  if (error.message.includes('敏感词')) {
    alert(error.message)
  } else {
    alert('发布失败：' + error.message)
  }
}
```

#### 业务中间件 - 敏感词过滤

[api/middleware/sensitiveFilter.ts](file:///Users/tog/Desktop/code/solo/xyj-120/api/middleware/sensitiveFilter.ts)

**核心功能：**
1. `filterSensitiveContent()` - 敏感词检测与替换函数
2. `sensitiveContentFilter` - 路由中间件，用于请求拦截
3. `checkSensitiveContent` - 独立接口处理器，供前端预检

**中间件使用方式（来自 [api/routes/community.ts](file:///Users/tog/Desktop/code/solo/xyj-120/api/routes/community.ts#L89)）：**

```typescript
// 在需要过滤的路由上挂载中间件
router.post('/', sensitiveContentFilter, (req, res) => {
  // 只有通过敏感词检查才会执行到这里
  // 否则中间件已经返回 400 错误
})

router.post('/:id/comments', sensitiveContentFilter, (req, res) => {
  // 评论同样需要过滤
})
```

**敏感词检查流程：**

```
请求到达 → 检查 body 中的 content/title/comment/text 字段
    ↓
检测到敏感词 → 返回 400 { success: false, error: '内容包含敏感词...', details: { foundWords, suggestion } }
    ↓
未检测到 → next() 继续后续路由处理
```

#### 全局错误处理中间件

```typescript
// 500 错误处理
app.use((error: Error, req: Request, res: Response, next: NextFunction) => {
  res.status(500).json({
    success: false,
    error: 'Server internal error',
  })
})

// 404 处理
app.use((req: Request, res: Response) => {
  res.status(404).json({
    success: false,
    error: 'API not found',
  })
})
```

### 5.1 开发模式

[package.json](file:///Users/tog/Desktop/code/solo/xyj-120/package.json) 中的脚本配置：

```json
{
  "scripts": {
    "client:dev": "vite",
    "server:dev": "nodemon",
    "dev": "concurrently \"npm run client:dev\" \"npm run server:dev\"",
    "build": "tsc -b && vite build",
    "preview": "vite preview"
  }
}
```

**开发流程：**
1. 运行 `npm run dev`
2. `concurrently` 同时启动两个进程：
   - Vite 开发服务器（端口 5173，默认）
   - Express API 服务器（端口 3001）
3. Vite 代理将 `/api/*` 请求转发到 Express

项目支持两种部署模式：

**本地开发部署：**
- 入口文件：[api/server.ts](file:///Users/tog/Desktop/code/solo/xyj-120/api/server.ts)

- 无状态函数执行

## 六、架构总结

### 6.1 优点

1. **清晰的分层架构**：前端组件 → API 封装层 → 后端路由 → 数据层，职责分明
2. **统一的接口规范**：前后端遵循一致的响应格式 `{ success, data, error }`
3. **类型安全**：全链路 TypeScript，前后端共享类型定义
4. **完善的错误处理**：前端统一抛出、后端多层拦截（参数校验 → 中间件 → 全局错误处理）
5. **良好的开发体验**：Vite 热更新 + Nodemon 自动重启 + 代理日志
6. **响应式设计**：从移动端到桌面端的完整适配

### 6.2 可改进点

1. **缺少请求/响应拦截器的扩展性**：当前 `request` 函数是硬编码的，如需添加认证 token、请求重试等功能需要修改核心函数
2. **缺少状态管理**：虽然使用了 Zustand，但当前页面数据都是各自管理，跨页面数据共享不便
3. **Mock 数据层**：当前是内存数组，后期接入真实数据库需要改造
4. **缺少加载状态统一处理**：每个页面单独管理 loading 状态，可考虑封装通用 Hook
5. **敏感词库硬编码**：可考虑配置化或接入第三方服务
